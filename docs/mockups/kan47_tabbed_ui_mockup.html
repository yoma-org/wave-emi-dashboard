KAN-47 — Multi-Attachment UI Mockup v13.3 proposal

Pattern-agnostic design: N attachments → N tabs + 1 Summary tab. Built by extending the existing single-attachment UI, not rebuilding it. Rendered in real Wave brand style so DK can eyeball before committing to code.

1. Pipeline flow (Worker v2 → v13.3)

Changes marked in yellow. Everything else is unchanged from v13.2.

Webhook (Worker Dispatch) → Claim & Reconstitute → Prepare for AI v3 → Count Attachments (new)
→ if >5: Reject & Send "too many" email → if ≤5: Split Attachments → Gemini Extract (per attachment) → AI Parse & Validate
→ Aggregate Results (new — builds attachments[] + summary) → Webhook POST to dashboard
Execution budget: Still 1 n8n execution per email. The loop lives inside a single Code node to stay within trial/starter cap.

2. Ticket detail modal — 3 attachments + Summary

Example: TKT-103, Capital Taiyo Co., Ltd. sends 3 attachments (1 payroll PDF + 1 employee list XLSX + 1 bank slip image). Whatever the pattern, the UI handles it the same way.

4. Rejection case — 6+ attachments

If client sends 6 or more attachments, pipeline rejects before creating a ticket and sends the client a helpful email. Dashboard never sees this email.

⚠ Auto-rejected email — no ticket created

From: finance@capitaltaiyo.mm · Subject: April payroll (many files) · Attachments: 7

Rejection email sent to sender:

Dear Sender,

We received your email, but our system can only process payroll requests
with up to 5 attachments per email. Your email had 7 attachments.

Please split your request into separate emails, each with no more than 5
attachments, and resend.

If you have questions, reply to this email.

Best regards,
Wave Money | E-Money Operations
Reuses existing rejection infrastructure: same Outlook node, same credential, same template scaffold as v12.2's empty-body rejection. New code = 1 extra branch in the "Is Rejection Email?" IF node.

6. Deferred questions (not blocking)

  • Smart summary rules — sum vs dedupe vs pick-one. Ship dumb summary, learn from real data.
  • Per-attachment approval — should Finance approve each tab separately? For now: 1 approval = whole ticket.
  • Partial failure — if 1 of 3 attachments is password-protected, process the 2 good ones and flag the 3rd. (Matches DK's original default.)
  • Tab naming — filename vs "Attachment 1". Default: filename if clean, else numbered.
