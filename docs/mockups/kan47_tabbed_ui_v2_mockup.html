KAN-47 — Multi-Attachment UI Mockup v2 · All-or-Nothing

Ships an "all-or-nothing" multi-attachment pipeline: all N attachments extract cleanly → one ticket; any fail → reject whole email with diagnostic message, no partial ticket. Adds email preview in Summary, full action matrix, Three-Way Match, rejection taxonomy, and system vs client failure routing.

What changed from v1

  • All-or-nothing processing — any failed attachment rejects the entire email (no partial tickets)
  • Email preview — collapsible panel in Summary tab only (removed from per-attachment tabs)
  • Three-Way Match block — email amount vs sum-of-attachments, explicit in Summary
  • Action matrix — explicit table of what each button does at ticket-level vs attachment-level
  • Failure taxonomy — 5 categories with reject-vs-retry routing
  • System-fail vs client-fail flow — retry silently on our errors, reject cleanly on theirs
  • 5 rejection email templates — one per failure category + combined multi-issue version
  • Removed "Retry this attachment" — all-or-nothing makes it unnecessary

1. Pipeline flow — v13.3 all-or-nothing

Evaluate all N attachments first, then gate. On ANY client-failure → reject whole email. On system-failure (our infra) → retry silently, don't penalize the client.

Pre-flight

Webhook (Worker Dispatch) → Claim & Reconstitute → Prepare for AI v3 → Count Attachments (new) → N > 5?
→ YES Reject: "too many attachments"

Loop (sequential, inside single n8n execution)

for each attachment i = 1 to N: Gemini Extract → AI Parse & Validate → Classify Outcome: pass / client-fail / system-fail

Gate after loop completes

Any system-fail?
→ YES Silent Retry (up to 3x with backoff) → then hard-fail to sys-admin → NO, continue
Any client-fail?
→ YES Reject whole email + diagnostic email listing ALL failed attachments → NO All pass → Aggregate → Create ticket → Webhook to dashboard
Execution budget: Still 1 n8n execution per email. The loop is a JS for-loop inside a Code node (not n8n's SplitInBatches) → stays on trial cap.
"Evaluate all" not "fail fast": even if attachment 2 fails client-side, we still try 3/4/5 before rejecting. That way the rejection email lists ALL issues at once → client fixes everything in one resend, no ping-pong.

Client sends 4 attachments. Attachment 2 is password-protected, Attachment 3 is an unreadable photo. Pipeline evaluates all 4, then rejects the whole email with a single diagnostic message listing BOTH failures.

Why evaluate-all not fail-fast: Notice the client gets both failures in one email. If we'd stopped at attachment 2, they'd fix it, resend, then discover attachment 3 is broken too, then fix that... ping-pong 3-4 times. Evaluate-all costs ~4¢ in wasted Gemini calls but saves the client multiple round-trips.
DB state after rejection: email_queue row status=rejected, full payload preserved for audit. tickets_v2 untouched. activity_log gets one entry: "Email from finance@xyz-corp.mm rejected: 2/4 attachments failed extraction".

5. Failure taxonomy & rejection email templates

Five outcome categories. Three trigger rejection (client's problem). One retries silently (our problem). One passes with a flag (marginal extraction, not a rejection).

CategoryDefinitionWho's at faultSystem action
HARD-FAIL: passwordZIP/PDF requires password we don't haveClientReject with template 1
HARD-FAIL: unreadableFile corrupted, 0-byte, or unsupported formatClientReject with template 2
HARD-FAIL: not payrollGemini confidence < 50% OR no extractable payroll fieldsClientReject with template 3
SYSTEM-FAIL: infraGemini 429/5xx, network timeout, Supabase downUsSilent retry 3x with backoff → escalate to sys-admin (NOT client)
SOFT-PASS: low confidenceGemini confidence 50-70%, extracted some fieldsNeitherProcess + flag on dashboard. Not a rejection.

6. System-fail vs client-fail routing

Critical correctness rule: we must NEVER reject a client's good email because Gemini had a bad minute. The classifier distinguishes infrastructure failures (our problem) from malformed inputs (client's problem).

Did extraction fail?
→
Yes
Error is 429 / 5xx / timeout / network?
→ YES
System-fail: silent retry up to 3x with exponential backoff (1s, 3s, 9s). If still failing after retries, escalate to sys-admin. Client sees nothing.
Error is password/corrupt/not-payroll/confidence?
→ YES
Client-fail: classify into one of templates 1-5. Collect all client-fails across the N attachments, send ONE diagnostic rejection email, mark email_queue.status=rejected. No ticket created.
Edge case: Mixed system-fail + client-fail on the same email. Rule: system-fail always takes priority. If the loop hits even one system-fail, retry the whole email. Only after all system-fails resolve do we classify client-fails and decide reject vs accept.
