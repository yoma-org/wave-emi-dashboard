KAN-47 — Multi-Attachment UI Mockup v3 · Side-by-Side · Human-in-Loop

Rendered using Nyan San Kim's real NSK disbursement email (2 attachments: payment list XLSX + bank slip PNG) provided by Vinh. Embraces the principle that this system's job is toil removal, not autonomous decisions. Summary = multi-column side-by-side, no aggregation, no clever inference.

Design principle

“AI Assists, Human Decides”

This system's job is to remove the repeated data-entry toil from bankers — copy, paste, manual typing, cross-checking numbers. It is not an autonomous disbursement system. Every decision (approve, reject, flag, return) still belongs to Finance. AI extracts, surfaces, and compares; humans decide.

Tenet 1
Toil removal, not automation
We remove data entry from bankers. We do not replace their judgment.
Tenet 2
Extract · Display · Compare
AI's three verbs. Never "decide," never "approve," never "infer intent."
Tenet 3
Side-by-side over aggregation
Consistency beats cleverness. One mental model for every ticket type.

What changed from v2

  • Added "Design Principle" banner — human-in-loop philosophy made explicit at the top of all future design docs
  • Summary tab = multi-column side-by-side — no aggregation, no dedupe, no "sum across attachments." Email column + N attachment columns. Finance eyeballs.
  • Killed the "not payroll" hard-fail rejection — bank slips and supporting docs are valid, not rejections. Fixes the NSK bug.
  • Killed `document_type` self-classification — unnecessary once we stop aggregating. Gemini just extracts what's there.
  • Three-Way Match = visual comparison, not an automated "PASS" assertion. Shows all 3 amount sources, Finance eyeballs alignment.
  • Simplified rejection taxonomy — 3 client-fail categories only: password / unreadable / too-many. No "confidence threshold" as rejection trigger.
  • CSV download with source-tab picker — banker selects which attachment's employee data becomes the Utiba CSV.
  • Uses real NSK email data from Vinh's Teams share — not synthetic Capital Taiyo.

1. Pipeline flow — v13.3 (simplified)

Fewer gates than v2. Loop over all attachments, extract whatever Gemini finds, reject only on infrastructure failure (our problem) or hard-fail categories (password/unreadable/too-many). "Not payroll" is no longer a rejection trigger — we trust Finance to eyeball.

Pre-flight

Webhook (Worker Dispatch) → Claim & Reconstitute → Prepare for AI v3 → Count Attachments (new) → N > 5?
→ YES Reject: template 3 "too many attachments"

Loop (sequential, inside single n8n execution)

for each attachment i = 1 to N: Gemini Extract (whatever's there) → AI Parse & Validate → Classify: pass / password / unreadable / system-error
Note v3 vs v2: No more "not payroll" classification. If Gemini can read the file, it's a pass — regardless of whether the content looks like payroll, a bank slip, a signature page, or anything else. Finance decides if it's useful.

Gate after loop

Any system-error? → YES Silent retry 3x backoff → sys-admin escalation
Any password/unreadable? → YES Reject whole email, diagnostic listing all failed files
Otherwise: All pass → Package attachments[] → Webhook → Dashboard renders tabs
Execution budget: Still 1 n8n execution per email. JS for-loop inside one Code node, not SplitInBatches.

Multi-column side-by-side: Email body on the left, Payment List (Tab 1) in the middle, Bank Slip (Tab 2) on the right. No sums. No dedupe. Finance eyeballs each row and sees where the numbers agree or diverge.

5. Rejection case — all-or-nothing, simplified taxonomy

Only 3 client-fail triggers reject a whole email: password-protected, unreadable/corrupted, or too-many attachments. "Not recognized as payroll" is NOT a rejection trigger anymore.

⚠ email_queue row status: rejected · no ticket created

Example: finance@xyz-corp.mm sends email with 3 attachments. Attachment 2 is a password-protected ZIP. Attachment 3 is a 0-byte corrupted image.

Per-attachment outcomes (all 3 evaluated before deciding)

#FilenameOutcomeReason
1payroll_apr.pdf✓ OKGemini extracted cleanly
2hr_data.zipFAILPassword-protected
3staff_photo.jpgFAIL0-byte file, corrupted

Rejection email sent to client (lists BOTH issues at once)

From: emoney@zeyalabs.ai To: finance@xyz-corp.mm Subject: Re: April Payroll — Unable to Process Dear Sender, We received your email with 3 attachments, but our system was unable to process it because 2 of the attachments could not be read. Specifically: ✘ hr_data.zip — The file is password-protected. Please resend without password or encryption. ✘ staff_photo.jpg — The file appears corrupted (0 bytes). Please check the original and resend. To process your request, please resend your email with ALL 3 attachments together, fixing the issues above. If you believe this rejection is in error, reply with Reference ID: QR-20260420-R19F and our team will investigate. Thank you, Wave Money | E-Money Operations
Evaluate-all, not fail-fast: Client gets all failures in one diagnostic email — no back-and-forth ping-pong. Costs ~4¢ in extra Gemini calls per rejected email, saves hours of client frustration.

7. System-fail vs client-fail routing

Critical rule: we must NEVER reject a client's good email because Gemini had a bad minute. The classifier distinguishes our infrastructure failures from client-side issues.

Did extraction fail?
→
Yes
Error is 429 / 5xx / timeout / network?
→ YES
System-fail: silent retry 3x with backoff (1s, 3s, 9s). If still failing, escalate to sys-admin. Client sees nothing.
Error is password / corrupt / too-many?
→ YES
Client-fail: classify into templates 1-3. Collect all client-fails, send ONE diagnostic rejection. email_queue.status=rejected. No ticket.
Otherwise (any readable content)?
→
Pass → render in tab. Finance decides if useful.
